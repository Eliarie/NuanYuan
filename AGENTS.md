# AGENTS.md：暖园 NuanYuan 项目 AI 开发指南

> 本文件是给 AI 智能体（如 Cursor, Claude, GitHub Copilot）阅读的系统提示词与架构约束。人类开发者请参阅 README.md。

## 1. 项目核心理念

本项目基于 **"Vibe Coding"（AI 辅助编程）** 理念开发。
所有参与本项目的 AI 代理（Agents）在生成代码、进行架构设计或业务决策时，**必须**严格遵循以下核心原则：

* **AI 友好**：代码结构、命名规范、接口设计必须符合 AI 训练数据中最常见的模式，确保 AI 能够理解、维护和扩展。
* **低复杂度**：如无必要，勿增实体。拒绝过度工程化，避免提前优化。
* **面向标准编程**：遵循行业通用标准（如 RESTful API 规范、标准的 HTTP 状态码、S3 API 兼容等），以便于 AI 更好的理解和后续的工具链接入。
* **无供应商绑定**：核心业务必须抽象出接口层，不强绑定于任何单一的云服务提供商或模型供应商。

**最终决策器**：优先生成简单、直接、可读性高的代码。如果遇到实现分歧，**永远选择最常见、最稳妥、AI 训练数据最多的方案**。

---

## 2. 项目描述与业务目标

* **项目名称**：暖园 · 家园共育系统 (NuanYuan)
* **核心定位**：基于 AI 的家园沟通与幼儿动态成长画像构建系统。通过“教师树洞”转化教师语音抱怨为规范的家园反馈，极大减轻教师工作量，同时让家长获得真实的在园动态。
* **MVP 目标**：跑通“教师吐槽 → AI处理与整理 → 安全推优给家长”的核心业务链路。

## 3. 技术栈字典与架构决策

### 3.1 项目拓扑结构 (Monorepo)

系统采用标准的 Monorepo 结构进行组织，便于前后端类型共享及统一构建：

```text
nuanyuan/
├── packages/
│   ├── shared/     # 核心类型定义、通用验证 Schema、多端枚举值
│   ├── client/     # 前端客户端 (微信小程序/H5/App)
│   └── server/     # 后端核心服务 (API 提供、大模型调度、数据库访问)
├── package.json    # 根项目配置
└── pnpm-workspace.yaml
```

### 3.2 核心技术栈与依赖

* **包管理器**: npm (基于 workspaces 的 monorepo)
* **语言**: TypeScript (Strict 模式)
* **前端框架**: Taro 4 + React 18。**跨端首选**：必须优先使用 Taro 提供的统一 API（如 `Taro.getStorage`），禁止直接使用单一平台特有 API（如 `wx.setStorageSync` 或浏览器 `window.localStorage`），最大化跨平台兼容性，避免跨端编译错误。
* **前端样式**: Tailwind CSS (通过 weapp-tailwindcss 适配小程序)。**限制**：避免使用过于复杂的 Arbitrary Values (任意值，如 `w-[100.5px]`)，尽量使用标准工具类以防止小程序类名转换编译失败。
* **前端 UI**: 自定义无头组件 (禁止引入 Taro-UI 或其他第三方大前端 UI 库)，所有的基础组件（如 Dialog, Popover）必须使用 Taro 原生的 <View>, <Text> 配合状态机手写实现。
* **设计风格**: 保持视觉系统的暖色调（奶白、暖黄搭配）与柔和感。必须应用大圆角 (rounded-2xl/3xl)、暖色低透明度阴影 (如 box-shadow: 0 10px 20px rgba(77,58,29,.09))，避免生成直角或冷硬的极简风格。
* **状态管理**: Zustand (严格限用于单纯的客户端 UI 状态管理，如弹窗、表单临时状态等)
* **数据请求**: TanStack Query (React Query v5) (负责且唯一负责 Server State。**关键适配**：小程序没有原生 fetch，必须将 `Taro.request` 封装为 Promise，并作为全局唯一的 `queryFn` 注入到 QueryClient 中。严禁在组件内使用 `useEffect` 裸写请求)。
* **后端运行时**: bun。**约束**：绝不在业务逻辑中使用 `Bun.file` 等 Bun 特有 API，必须使用标准 Node.js 模块 (如 `fs/promises`) 或标准 Web API，以保证随时无痛降级或切换至 Node.js 运行时。
* **后端框架**: Hono
* **数据库 ORM**: Prisma (连接 PostgreSQL)
* **全栈验证**: Zod
* **认证方案**: hono/jwt + 微信小程序原生 code 登录
* **日志系统**: pino + hono-pino

### 3.3 核心中间件与抽象层设计

为贯彻“无供应商绑定”和“低复杂度”理念，必须在代码层面落实以下适配器模式：

1. **媒体与文件存储机制（存储适配器模式 - Storage Adapter Pattern）**
   * **初期策略**：使用简单的**本地文件系统存储**记录小程序上传的录音/文件（直接存入指定目录即可）。
   * **代码要求**：业务逻辑不能直接调用由 OS 提供的读写 API，必须注入 `StorageService` 接口，当前实现为 `LocalStorageProvider`。接口需严格兼容标准的 S3 API 设计（上传、获取 URL、删除），以便未来一键热切换迁往腾讯云 COS / 阿里云 OSS。

2. **多通道消息通知机制（通知推送服务层 - Notification Service Layer）**
   * **设计剥离**：不要将系统硬绑定于“微信小程序订阅消息”。
   * **扩展架构**：封装统一的 `NotificationService`。上游业务仅需下发生存通知要求，由服务层路由推送策略（目前核心支持 `WeChatMiniProgramChannel`，并预留 `AppPushChannel` 和 `WebhookChannel` 接口）。

3. **通用大模型接口层 (LLM Gateway)**
   * **开放模型调度**：不要硬编码指定某个特定型号的 AI（如不限定 DeepSeek / Kimi）。
   * **标准对接**：建立全局的 `LLMService` 层。采用业内成熟且强大的 **Vercel AI SDK (`ai` npm package)** 进行大语言模型调度。该库具备良好的系统设计，要求利用核心方法（如 `streamText`/`generateText`）配合其内置的 `CoreMessage` 数组来进行对话及跨会话上下文生命周期管理。这将允许团队无缝穿插调度任何兼容标准的模型。
   * **跨端流式对话通信**：在后端（Hono），必须将 `streamText` 结果转换为标准 Web Streams API 输出；在前端（小程序），严禁使用标准 Fetch API 接收流，**必须使用 `Taro.request` 且配置 `enableChunked: true` 来接收后端的流式数据**，并手动解析 chunk 完成打字机效果。
   * **多步链路拆分 (LLM Pipeline)**：业务具有“一边提供安抚响应，一边后台静默提取数据”的诉求。必须将两者独立调用：对话使用 `streamText`，结构化数据提取（生成推优/提取个案）使用 `generateObject` 结合 Zod Schema 保证输出稳定结构。严禁让大模型在一次对话请求中既输出情感话语又返回 JSON 字符串。

### 3.4 部署与运行环境 (Deployment)

本项目采取跨端渐进式发布与混合部署策略：

* **端侧运行目标**：
  * **开发与测试**：优先使用 Web H5 形式进行快速调试与功能验证。
  * **初期（MVP）**：核心依托于**微信小程序**运行。
  * **未来规划**：基于 Taro 的跨端能力，后期将扩展至原生 App。
* **前端发布**：
  * **H5 端**：部署在 **Vercel** 上，实现敏捷的 CI/CD 集成。
  * **小程序端**：通过**微信开发者工具**进行编译构建与提审上传。
* **后端部署**：
  * 采用 **Docker Compose** 统一编排后端服务（Bun 运行时 + 业务 API）与底层依赖（PostgreSQL 数据库）。
  * 将容器化后的产物部署在**个人云服务器**上，保持架构的高度独立性。

## 4. 绝对禁令 (DO NOT)

当 AI 代理在处理本项目时，**绝对禁止**以下操作（触发则视为重大违背）：

1. **严禁过度设计**：禁止引入如微服务架构、Kafka、K8s 等复杂的大型分布式组件（我们处于 MVP 阶段，模块解耦靠代码抽象而非物理隔离）。
2. **严禁错误的环境指令**：后端严禁生成使用 `yarn` 或 `pnpm` 启动及安装的指令；全局依赖严禁破坏 `npm` monorepo 约束。
3. **严禁硬编码三方密钥**：任何 API Keys（微信密钥、大模型 Token、数据库 URL 等）绝对不能写死在代码中，必须从 `.env` 中提取，且强检验环境变量配置。
4. **严禁数据越权流出**：教师端的“AI 树洞”原始情感倾诉（抱怨/吐槽文本）、初步转录文本等核心素材，**绝对不可**被封装至拉取给家长端查看的 API 响应体中。必须确保响应报文已被“端点数据切面 (DTO)”清洗。**数据库设计层面，必须物理隔离，将“树洞对话记录 (RawRecord)”与“家长可见记录 (SharedRecord)”设计为独立数据表**。同时，必须支持多对多关系模型：一个幼儿园包含多班级，一个班级关联多名教师和多名学生，一个学生可绑定多名家长（支持不同角色如爸爸、妈妈等）。在记录 `RawRecord` 时，必须关联具体的教师 ID，以便 AI 综合多名教师的视角；在查询 `SharedRecord` 时，需根据家长绑定的学生 ID 进行权限控制。
5. **严禁直接绑定云厂商 SDK**：不允许业务代码最深处直接 `import { COS } from 'tencentcloud-sdk'`，必须走到我们自己设计的 `StorageService` 中。
6. **严禁处理多媒体/设备 API**：MVP 第一版**仅支持纯文本交流（语音输入完全交由用户手机自带输入法的语音转文字功能实现即可）**。严禁实现任何语音、图片等多媒体文件的上传、处理或预览。绝对禁止调用原生的 Web/DOM 媒体 API（如 `navigator.mediaDevices` 或 `window.MediaRecorder`），也暂不使用小程序的录音和相册 API（忽略且不实现原型中的麦克风功能）。

## 5. 开发工作流指南

1. **分析需求（Ask / Contextualize）**：AI 在接收到新需求时，先分析属于前端 (client) 还是后端 (server)。
2. **规划步骤（Plan）**：始终分步骤进行，不要在一个响应中强行生成跨越数个系统全量代码。
3. **接口为先（Interface First）**：如果涉及前后端交互，优先在 `packages/shared` 里制定 TypeScript 接口或 Zod Schema。
4. **实现与验证**：按照 `Storage Adapter` 和 `LLM Gateway` 的指导进行依赖抽象与编码。
